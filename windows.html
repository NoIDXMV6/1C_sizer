📊 Запуск теста Гилёва (TPC-1C) на Windows

Тест измеряет производительность сервера в условных операциях в секунду. Запустите на сервере 1С (не на клиенте):

# PowerShell: скачать и распаковать
Invoke-WebRequest -Uri "http://www.gilev.ru/tpc1cgilv/tpc1cgilv.zip" -OutFile "C:\tpc1c.zip"
Expand-Archive -Path "C:\tpc1c.zip" -DestinationPath "C:\tpc1c"
# Запустить tpc1c_gilv.exe из папки C:\tpc1c

Интерпретация:
• < 15 оп/с – очень низкая производительность, требуется апгрейд.
• 15–30 – средняя, возможны задержки при 20+ пользователях.
• 30–50 – хорошая, комфортная работа.
• >50 – отличная, подходит для высоких нагрузок.
Важно: тест однопоточный и чувствителен к тактовой частоте CPU и схеме электропитания. Переключите план питания в «Высокая производительность» перед тестом. В виртуальных машинах результаты могут быть ниже реальных.

⚙️ Подбор сервера на планируемую нагрузку
💡 Расширенные рекомендации (Windows + MS SQL Server)
• Антивирус: Добавьте каталоги данных SQL Server (.mdf, .ldf, .ndf), рабочие каталоги 1С, а также tempdb в исключения. Сканирование этих файлов в реальном времени критически снижает производительность.
• tempdb: Разместите tempdb на отдельном NVMe-диске. Создайте несколько файлов данных tempdb (рекомендуется: min(количество ядер CPU, 8) файлов одинакового размера). Включите флаг трассировки TF 1117 и TF 1118.
• Max Server Memory: Всегда ограничивайте. Оставьте для ОС: 4 ГБ при ≤16 ГБ RAM, 10% при >32 ГБ (минимум 4 ГБ). Пример: при 64 ГБ RAM → Max Server Memory = 57 ГБ.
• MAXDOP (степень параллелизма): Для OLTP-нагрузки 1С установите MAXDOP = 1 или 2. 1С плохо параллелит запросы, и высокий MAXDOP может привести к деградации производительности. Рекомендуем MAXDOP = 1, при необходимости – 2. Cost Threshold for Parallelism = 50.
• Регламентные задания: Обслуживание индексов (reorganize/rebuild), обновление статистики, резервное копирование – только в нерабочее время через SQL Server Agent.
• Дисковая подсистема сервера приложений: Для сервера 1С достаточно SSD (SATA) объёмом от 100 ГБ. RAID 1 – для отказоустойчивости. Производительность диска на этом узле не критична, основная нагрузка на ввод-вывод ложится на SQL Server.
• Разделение серверов: При 30+ пользователях разделите сервер 1С и SQL Server. При 80+ – кластер серверов 1С (2+ узла).
• Высокая доступность: AlwaysOn Availability Groups – для чтения/записи (не требует shared storage). FCI – для отказоустойчивого экземпляра (требует shared storage: SAN, S2D).
